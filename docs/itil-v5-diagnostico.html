Diagnóstico ITIL v5 · Maturidade Tática

SpecDriven ITSM — onde já somos ITIL, e o que falta pra fechar

O produto já implementa, na camada de código e no fluxo de UI, um subconjunto expressivo das práticas de gestão de serviços do ITIL v5 — em particular toda a família de suporte a usuário (Incidente, Requisição de Serviço, SLA, Service Desk) e o esqueleto de Service Request Fulfillment. Há também um embrião maduro de gestão de mudanças (Change Enablement via aprovações) e de catálogo (Service Catalog) já parametrizável pelo gestor.

O gap real para maturidade tática (visível + métricas) não é reescrever nada: é (a) rotular o que já existe com o vocabulário ITIL v5, (b) expor métricas calculadas em dashboard, (c) fechar três práticas que estão semi-prontas mas não-acionáveis (Problem Management, Change Enablement formal, Knowledge Management) e (d) introduzir um Catálogo de Serviços com SLA por oferta — não só por cliente.

3. Práticas fortes — o que já é sólido, com prova em código #

A qualidade do que já existe é o motivo principal pra escrever este diagnóstico: implementar ITIL v5 num SaaS B2B2C do porte desse tipicamente leva 6–12 meses. Vocês já estão muito à frente. Esta seção mostra a evidência.

3.1 Incident Management — fluxo completo, com cálculo de SLA

O tipo ITIL incidente é uma das quatro TicketType e tem ciclo de vida próprio (backlog → em_andamento → aguardando_cliente → em_teste → concluido / cancelado). O primeiro atendimento é automático:

// apps/api/src/tickets.ts (linhas ~340-350) if (statusChanging && nextStatus) { if (!ticket.firstResponseAt && previousStatus === "backlog" && nextStatus !== "backlog" && nextStatus !== "cancelado") { slaExtras.firstResponseAt = new Date(); } if (nextStatus === "concluido" || nextStatus === "cancelado") { slaExtras.resolvedAt = new Date(); } else if (previousStatus === "concluido" || previousStatus === "cancelado") { slaExtras.resolvedAt = null; slaExtras.slaDueAt = await computeSlaDueAt({ ... from: ticket.createdAt }); } }

E o cálculo de SLA é um motor de horas úteis completo (apps/api/src/sla-calc.ts), com suporte a horário comercial por política, feriados por organização, e janela [start, end) por dia da semana ISO (1=Seg…7=Dom). É ITIL-bulletproof.

3.2 Service Level Management — alvo de 90% no overview

O packages/shared/src/sla-helpers.ts já entrega três períodos de análise (mês atual, mês anterior, trimestre) e a função computePeriodSlaPct é puro, testável, e usada tanto no overview staff quanto na home do cliente.

3.3 Service Desk — dois portais, sino, busca, comentários, visibilidade

Modelo de comentário carrega visibility: public | internal (essencial ITIL — comentários internos não viram notificação do cliente). Notificações usam matriz evento×canal configurável por consultoria (defaultNotificationPrefs()).

3.4 Service Catalog — habilitação por consultoria

Catálogo configurável em /settings/catalog: quatro tipos ITIL (melhoria, incidente, duvida, problema) + módulos customizáveis (TicketModuleCatalog com chave, label, sortOrder, enabled). O portal cliente consome via GET /portal/settings — o que é a essência de Service Catalog Management.

3.5 Audit trail — base sólida para conformidade

AuditEvent é append-only e cobre todas as ações sensíveis (settings, mudanças, etc.). O handler de escrita é fail-safe: falha de audit não quebra o fluxo principal (apps/api/src/audit.ts:23).

// apps/api/src/audit.ts try { await prisma.auditEvent.create({ ... }); } catch { // Audit must not break primary flows. }

3.6 Change Enablement (parcial) — workflow de aprovação reutilizável

ApprovalRequest já cobre três tipos: ticket (ex. concluir), hour_limit (ampliar), time_entry (lançamento). Aprovações são transacionais, idempotentes, e com decision note. É a base certa para evoluir pra RFC formal.

3.7 Demand & Capacity (parcial) — baseline por cliente

O conceito de baseline (horas contratadas/mês) já é primeira-classe: Client.baselineHoursMonth, flag countsTowardBaseline por ticket, e endpoint getBillingSummary(clientId, from, to) que retorna baselineRemaining — exibido no overview (KPI "Baseline restante").

4. Gaps por severidade #

Severidade considera (a) impacto operacional se não-resolvido, (b) frequência de uso, e (c) o que o mercado B2B2C tipicamente exige como barreira de entrada. Critical = bloqueia maturidade tática · High = precisa entrar no próximo trimestre · Medium = polish tático · Low = nice-to-have.

Critical — bloqueia maturidade tática

C1. Sem dashboard de CSI / métricas táticas (MTTR, MTTA, FCR, change success, baseline burn, backlog aging)

Hoje o OverviewPage mostra: SLA cumprido, baseline restante, horas pendentes, chamados ativos. Falta um painel "Saúde do Serviço" com:

  • MTTA (Mean Time To Acknowledge) — firstResponseAt − createdAt em horas úteis.
  • MTTR (Mean Time To Resolve) — resolvedAt − createdAt em horas úteis.
  • FCR (First Call Resolution) — % de tickets que saíram de backlog direto para concluido sem voltar pra em_andamento.
  • Change success rate — % de aprovações approved vs rejected por kind.
  • Baseline burn — soma de horas aprovadas vs baseline no mês, por cliente.
  • Backlog aging — distribuição de idade de tickets abertos por bucket (1d, 3d, 7d, 30d).

Onde mora hoje: apps/api/src/reports.ts só agrega por status. O cálculo está espalhado em sla-helpers.ts e no OverviewPage. Falta uma API GET /reports/service-health?period=....

C2. Change Enablement é só "aprovar para concluir" — não é RFC

ITIL v5 separa "Change Enablement" (RFC com risco, plano, janela) de "Change Control" (gatekeeping). Hoje ApprovalRequest resolve só o lado do gatekeeping. Falta:

  • Entidade Change com riskScore, rollbackPlan, windowStart/End, cabDecision.
  • Vinculação Change → Ticket afetados (para mudar uma config, afeta N chamados).
  • Vinculação Change → Approvals já existentes.
  • Dashboard "Change Calendar" (semelhante ao Overview, mas com timeline de mudanças).

Arquivos a tocar: schema.prisma (modelo Change), apps/api/src/ (novo changes.ts), apps/web-staff/src/pages/ (nova ChangesPage.tsx).

C3. Problem Management é só um label — não há entidade Problem nem Known Error

TicketType.problema existe mas se comporta idêntico a incidente. Não há:

  • Modelo Problem (id, title, rootCause, workAround, status investigating / identified / known_error / closed).
  • Vínculo Incident ↔ Problem (N:N).
  • Base de Known Errors (KEDB) — artigos com root cause, workaround, e link pra Defect que originou.
  • UI staff /problems com KEDB e link pro ticket.

Por que é Critical: sem Problem, vocês vão pagar caro em incidentes repetidos. Em modelo B2B2C com 1-2 consultores por cliente, o ticket-problema é a única forma de registrar "causa raiz conhecida" e virar item da base de conhecimento.

C4. Knowledge Management é só "link externo"

knowledgeBaseUrl aponta pra fora. Pra B2B2C isso é subótimo: (a) tira a marca da consultoria, (b) KB não tem contexto do ticket, (c) o que é "Knowledge" no ITIL inclui artigos vinculados a tickets, problemas, e procedimentos.

Esboço de solução: entidade Article com title, body (markdown), category, linkedTicketKeys (M:N), linkedProblemIds (M:N), visibleToClient, e workflow de publicação (rascunho → publicado).

High — entra no próximo trimestre

H1. Service Catalog precisa virar "Service Offering" (entidade)

Hoje o catálogo é uma enum + módulo. ITIL v5 fala em Service Offering: nome, descrição, prazo (SLA), custo, aprovação requerida, status (ativo/em construção/descontinuado).

Modelo proposto: ServiceOffering com FK organizationId, name, description, slaPolicyId (referência a SlaPolicy), requiresApproval, status, version. Substitui a enum TicketType na criação de tickets.

H2. Sem Major Incident process

Quando um incidente crítico paralisa um cliente (ex: sistema fora), ITIL exige Major Incident: comunicado, war room, post-mortem. Hoje priority: "critica" só muda a cor.

Esboço: flag isMajor em Ticket, template de "comunicado de major" (e-mail em massa pros stakeholders do cliente), página dedicada /majors com timeline.

H3. Backlog aging visível

"Chamados ativos" hoje não diz quais estão envelhecendo. Falta um widget de aging por bucket (0-1d, 1-3d, 3-7d, 7-30d, 30d+) com drill-down e link pra "stale" (sem update há N dias).

H4. Sem workflow de "Problem → Change"

Quando um Problem é identificado e precisa de um fix de infraestrutura, deve virar uma Change com janela, CAB, etc. Sem essa ligação, vocês têm "Problema" + "Change" como silos. ITIL chama isso de "problem-to-change workflow".

H5. Visibilidade ITIL pros consultores

O portal cliente tem o painel "Acompanhamento" (chamados abertos, SLA do mês, última atualização) — bom. Mas falta do lado staff o painel "Saúde do Serviço" (C1) e o "Calendário de Mudanças" (C2). Hoje a navegação staff não tem nenhuma página dedicada a visão ITIL.

Medium — polish tático

M1. CSAT pós-resolução

Cliente fecha o chamado, recebe e-mail "Como foi? 1-5 ⭐" com link pra rota /tickets/{key}/feedback?score=N. Hoje nada. Notification + Ticket (campo csatScore + csatComment).

M2. Exportação CSV/Excel dos relatórios

ticketsReport retorna JSON. Pra PMO, é CSV/Excel. GET /reports/service-health.csv?period=....

M3. SLA warning por e-mail (já no P2 do docs)

Faltam apenas dois eventos na matriz NotificationPrefs: ticket.sla_warning (ex: 80% do prazo consumido) e ticket.sla_breached. Scheduler simples (cron a cada 15 min) que varre tickets com slaDueAt próximo.

M4. Recalcular SLA em massa já existe; falta versionar

recalculateOpenSlaHandler existe em apps/api/src/sla.ts:347. Falta versionar a alteração (audit) e mostrar a UI "Última recálculo: ..." no /settings/sla.

M5. SLA "por service offering" (não só por cliente)

Hoje: SLA por (cliente, prioridade). ITIL sugere SLA também por tipo de serviço. Ex: "Incident = 4h úteis; Service Request = 1 dia útil; Problem = 5 dias úteis". Isso muda a forma de contratar.

Low — nice-to-have

L1. Risk register (entidade Risk)

Risk com score, mitigation, owner, status. Útil para board.

L2. NPS trimestral

Similar ao CSAT, com cadência maior.

L3. Capacidade (Capacity & Performance)

Forecast de horas vs baseline no próximo trimestre.

L4. Uptime do próprio portal

Pra vender SLA de "99,5% do portal no ar", é preciso medir. UptimeRobot free ou BetterStack.

L5. Continuar com o JWT em localStorage é risco XSS

Documentado em docs/security.md. Mover pra httpOnly + SameSite=Lax cookie. Não é ITIL puro, mas é blocker de Information Security Management.

5. Roadmap priorizado — 4 ondas #

Cada onda é um ciclo autônomo, com entregável claro. Esforço é em dias-pessoa (DP) considerando o time atual (1-2 pessoas, conforme observado em user_profile e na estrutura do monorepo).

Onda 0 — Quick wins táticos (1-2 semanas · 4-6 DP)

Sem mudança de schema. Toca em UI + endpoints já existentes.

  • QW1. Endpoint GET /reports/service-health com MTTA, MTTR, FCR, change success, baseline burn, backlog aging (3-4 DP)
  • QW2. Nova página staff /reports/health consumindo QW1 com cards/gráficos (2 DP)
  • QW3. CSAT pós-resolução: e-mail + 1 endpoint POST /tickets/{key}/feedback + campo csatScore no schema (1-2 DP)
  • QW4. Aging widget no OverviewPage com drill-down (1 DP)

Entregável: dashboard "Saúde do Serviço" + CSAT ao vivo.

Onda 1 — Problem + Change (4-6 semanas · 12-18 DP)

Schema novo. Modelagem cuidadosa. É a fundação ITIL v5 madura.

  • W1.1. Modelo Problem + migração + Incident.problemId (3 DP)
  • W1.2. API REST /problems + UI staff /problems (3 DP)
  • W1.3. Modelo Change (RFC) com risk, window, rollback, CAB (3 DP)
  • W1.4. API REST /changes + UI /changes + Change Calendar widget (4 DP)
  • W1.5. Workflow "Problem → Change" — botão no Problem que cria Change com campos pré-preenchidos (2 DP)

Entregável: Problem Management e Change Enablement vivos, com linkage Problem↔Change↔Incident.

Onda 2 — Knowledge + Service Offering (4-6 semanas · 10-15 DP)

Substitui a enum TicketType por entidade. Cuidado com migração.

  • W2.1. Modelo Article (markdown) + estado draft / published (3 DP)
  • W2.2. UI staff /knowledge (lista, busca, edição) + UI cliente embutida no detalhe do ticket (3 DP)
  • W2.3. Modelo ServiceOffering + migração (deixar TicketType como legado por 1 release) (3 DP)
  • W2.4. UI staff /settings/catalog reformulada: lista de Service Offerings em vez de toggles (3 DP)
  • W2.5. Portal cliente renderiza catálogo público de offerings (2 DP)

Entregável: KEDB interna + catálogo de serviços formal com SLA por oferta.

Onda 3 — Maturidade tática contínua (ongoing · 2-3 DP/mês)

Não é projeto, é prática. CSI + security + compliance.

  • W3.1. SLA warning/breach por e-mail (scheduler +2 eventos na matriz) — mover do P2 docs (2 DP)
  • W3.2. Export CSV/Excel dos relatórios (1 DP)
  • W3.3. Migrar JWT pra cookie httpOnly + SameSite=Lax (1-2 DP)
  • W3.4. Risk register (entidade Risk) — opcional mas valioso pra board (2 DP)
  • W3.5. Revisão trimestral de CSI: atas, action plan, retro (processo, não código)

Entregável: cadência tática ITIL v5 mantida.

Recomendação de início: Onda 0 (quick wins) traz visibilidade imediata com baixíssimo risco — é onde o ROI aparece em 1-2 semanas. Onda 1 (Problem + Change) é o "salto de maturidade" que diferencia SaaS amador de ITSM sério. Onda 2 é o que transforma vocês em plataforma, não só ferramenta.

9. Riscos e anti-padrões a evitar #

R1. Implementar Problem Management sem Change Enablement (vice-versa)

Os dois precisam nascer juntos. Sem Change, Problem vira "ticket de reclamação eterna". Sem Problem, Change vira "deploy sem causa raiz". Implementem na Onda 1 inteira.

R2. Migrar enum TicketType para ServiceOffering sem faseamento

Tem cliente com 200 tickets legados usando incidente. Quebre em duas releases: (1) adiciona ServiceOffering sem remover a enum; (2) migra formulário; (3) depreca a enum como filtro de "legacy" no relatório.

R3. Knowledge com edição livre pelos consultores sem curadoria

KEDB sem workflow vira wiki abandonada. ITIL recomenda Knowledge Curator. Implemente status: draft / published desde o dia 1, com aprovação do gestor antes de virar visível ao cliente.

R4. Excesso de métricas (vanity metrics)

Métrica que não muda decisão é ruído. As 4 famílias do cap. 6 cobrem o que importa taticamente. Resistam a adicionar "ticket médio por hora do dia" — não leva a ação.

R5. "Mudar pra virar ITIL" sem onboarding do cliente

Cliente acostumado a "abrir ticket" pode estranhar o "Service Catalog com offerings". ITIL v5 é uma promessa de transparência, mas a UX precisa continuar simples. O catálogo novo entra atrás do botão "Abrir chamado" — não substitui.

R6. Esquecer de auditar as novas entidades

Toda ação de escrita em Problem, Change, Article deve gerar AuditEvent. Vocês já têm o padrão; basta replicar o padrão de writeAudit(...) dos handlers de settings.ts.
